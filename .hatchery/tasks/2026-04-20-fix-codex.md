# Task: fix-codex

**Status**: complete
**Branch**: hatchery/fix-codex
**Created**: 2026-04-20 11:38

## Objective

Fix codex API key: container was sending 401 Unauthorized to `api.openai.com/v1/responses`
with an incorrect key (the proxy UUID token), and the proxy was never involved.

## Context

User auth: `~/.codex/auth.json` with `auth_mode: "chatgpt"`, `OPENAI_API_KEY: null`,
and valid `tokens.access_token` (OAuth JWT). No `OPENAI_API_KEY` env var.

The 401 error showed a UUID-format key (`91d78902...6201`) — matching the proxy token
format — being sent directly to `api.openai.com`.

## Summary

**Root causes** (two separate issues):

1. `_read_codex_creds()` did not check `auth_mode` before looking at `OPENAI_API_KEY`.
   If a stale `OPENAI_API_KEY` env var was present in the hatchery process, it would
   shadow the user's intentional OAuth login, causing the proxy to target
   `api.openai.com` instead of `chatgpt.com`.

2. codex (≥ v0.121) ignores the `OPENAI_BASE_URL` environment variable entirely.
   The only supported mechanism for a custom base URL is `openai_base_url` in
   `config.toml` or `--config openai_base_url='"..."'` at the CLI. Setting
   `OPENAI_BASE_URL` in `container_env` had no effect, so codex always called the
   hardcoded `api.openai.com` endpoint (and sent the proxy UUID token there).

**Fixes in `src/seekr_hatchery/agents/codex.py`**:

1. `_read_codex_creds()`: check `auth_mode` first. When `auth_mode` is "oauth" or
   "chatgpt", skip `OPENAI_API_KEY` (env var and file field) and use
   `tokens.access_token` directly. This ensures the host correctly detects OAuth mode
   and points the proxy at `chatgpt.com` rather than `api.openai.com`.

2. `on_before_container_start()`: always write `auth_mode: "apikey"` in the fake
   auth.json, regardless of the host's real auth mode. In apikey mode, codex respects
   `OPENAI_BASE_URL` (if it did), and crucially uses the proxy URL from --config.
   In chatgpt mode, codex bypasses `OPENAI_BASE_URL` entirely.

3. `build_new_command` / `build_resume_command` / `build_finalize_command`: when
   `docker=True`, wrap the codex invocation in `sh -c` that reads `$OPENAI_BASE_URL`
   (already set by `container_env`) and passes it to codex via `--config`:
   ```
   exec codex --config "openai_base_url=\"$OPENAI_BASE_URL\"" \
       --dangerously-bypass-approvals-and-sandbox "$@"
   ```
   The prompt is passed as a positional arg (`"$@"`), preserving all whitespace
   and special characters. Native mode is unchanged.

**Critical invariants for future agents**:

- The fake auth.json must always use `auth_mode: "apikey"` — only apikey mode routes
  through `OPENAI_BASE_URL`. chatgpt mode bypasses it and goes directly to chatgpt.com.
- codex ignores `OPENAI_BASE_URL` env var. Always inject via `--config` CLI flag.
- The routing to the correct upstream (api.openai.com vs chatgpt.com) is handled on
  the host side by `proxy_kwargs()`, not inside the container.
