# Task: fix-from

**Status**: complete
**Branch**: hatchery/fix-from
**Created**: 2026-03-19 15:33

## Objective

When starting hatchery with `--from`, and the `dockerfile/config` do not exist, we get an annoying error:

- Hatchery will detect the missing dockerfile _from our current branch_
- It will ask to add the dockerfile, which will be added _to our current branch_
- it will then spawn a task from the `--from` branch, which will have no dockerfile!

We need to fix hatchery so:
- It should always look at the branch it is STARTING from to determine if the dockerfile exists or not
- When it creates the dockerfile and commits it, it should commit it to the _worktree branch_ (when worktree is used)
  - The best way to do this is probably to create the worktree _first_, then check, populate,etc
  - If the task is abandoned before startup (e.g. Ctrl-c out of it), we can cleanup the worktree that is created

## Context

The bug was in `cmd_new()` in `src/seekr_hatchery/cli.py`. The original order was:
1. Check/create dockerfile + docker config (against `repo` root, current branch)
2. Commit to current branch
3. Create worktree from `--from` branch

The worktree forked from `--from` never received the Dockerfile, so `resolve_runtime()` failed with "No Dockerfile found".

## Summary

**Fix**: Reordered `cmd_new()` in `src/seekr_hatchery/cli.py` so worktree creation happens before dockerfile/config checks.

**New order**:
1. `ensure_tasks_dir` + `ensure_gitignore` (repo)
2. Name resolution and conflict check
3. Create worktree (if `not no_worktree`)
4. **All subsequent steps in a `try/except KeyboardInterrupt` block:**
   - `ensure_dockerfile(worktree, ...)` + `ensure_docker_config(worktree)` — now using worktree path
   - Commit dockerfile from `cwd=worktree`
   - Editor or objective prompt
   - Commit task file
   - Save meta, resolve runtime, launch

**Key decisions**:
- `worktree = repo` when `no_worktree=True`, so the `no_worktree` path is identical to before
- The `try/except KeyboardInterrupt` wraps all post-worktree steps (not just dockerfile creation), matching the editor-cancel cleanup pattern that already existed
- `git.remove_worktree` + `git.delete_branch` are called on interrupt if `not no_worktree`

**Gotcha**: `dockerfile_path(worktree, backend).relative_to(worktree)` and `DOCKER_CONFIG` paths must be relative to `worktree` for the `git add` to work correctly inside the worktree.
