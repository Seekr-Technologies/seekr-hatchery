# Task: no-commit

**Status**: complete
**Branch**: hatchery/no-commit
**Created**: 2026-07-10 09:05

## Objective

We need to update hatchery so we have the option to not commit any hatchery related files
(task files, Dockerfiles, `docker.yaml`, etc.), and to make that a persistent default rather
than a per-invocation flag. The challenge: if hatchery no longer commits these files, we need a
robust way for the user to *manage* their Dockerfile(s) and task files per repo without losing
them and without leaving hatchery artifacts scattered in the tracked repo.

## Context

### Current state (before this task)

Partial no-commit support already existed (built by `2026-04-22-no-commit.md` and
`2026-04-20-fix-no-commit-docker.md`):

- **CLI flags only:** `--no-commit` on `new`/`chat`/`sandbox`; `--no-commit-docker` on `new`.
  There was **no persistent config** — you had to pass the flag every run.
- **`--no-commit` behavior:** Docker files were written to the repo root and copied into the
  worktree (uncommitted). The task file was written into the worktree at
  `<worktree>/.hatchery/tasks/<date>-<name>.md` and never staged.
- **`no_commit` was persisted** on `SessionMeta.no_commit` so `done`/`archive` skip their
  checkpoint-commit prompts.

### Problems this task fixed

1. **No persistent knob.** No-commit was per-run only; no per-user default.
2. **Task-file data loss.** In no-commit mode the task file lived only in the worktree, so
   `hatchery done` (which removes the worktree) destroyed it.
3. **Repo pollution.** Uncommitted `.hatchery/Dockerfile.*` / `docker.yaml` sat as untracked
   files in the tracked repo.
4. **`--no-commit-docker` was a confusing separate axis.** It split docker-file location from
   task-file location, creating a confusing matrix. Dropped entirely — docker files always share
   the task file's commit mode.

## Summary

### Key decisions

- **Single no-commit axis.** `SessionMeta.no_commit: bool = False` (the released field name).
  `no_commit_docker` is deleted entirely with no migration. Two modes:
  - **Committed mode** (default, `no_commit=False`): task file + `Dockerfile.<agent>` +
    `docker.yaml` under `<worktree>/.hatchery/…`, committed on the task branch.
  - **Not-committed mode** (`no_commit=True`): both under
    `~/.hatchery/repos/<repo-id>/{records,docker}/`; worktrees hidden via `.git/info/exclude`;
    nothing hatchery in the tracked repo.
- **Config knob:** `auto_commit: bool = True` in `~/.hatchery/config.json`. `false` makes
  no-commit the default for every repo.
- **Tri-state CLI flag:** `--commit/--no-commit` (default `None`) on `new`, `chat`, and
  `sandbox`. Resolution: `no_commit = (not commit) if commit is not None else (not cfg.auto_commit)`.
  The no-commit mode is **pinned on the task** at `new`/`chat` time via `SessionMeta.no_commit`.
- **`--no-commit-docker` flag dropped entirely.** Docker files always share the task file's
  mode; there is no docker-only axis.
- **Everything out-of-tree in no-commit mode:** Both Docker files and task-file records live
  under `~/.hatchery/repos/<repo-id>/`. Nothing hatchery-related is written into the tracked repo.
- **Record store = plain files now, git-ready layout.** The layout
  (`~/.hatchery/repos/<repo-id>/records/YYYY-MM-DD-<name>.md`) is chosen so it can become an
  out-of-band git repo later, purely additively.
- **Local ignore:** The worktree dir (`.hatchery/worktrees/`) is hidden via `.git/info/exclude`
  (local, uncommitted) in no-commit mode — never the committed `.gitignore`.
- **Record store mounted RW:** In no-commit mode, `build_mounts` adds a RW bind-mount of
  `meta.record_dir` so the agent edits the live record and can read sibling records for
  cross-task context.
- **No schema migration.** `SCHEMA_VERSION = 1`, `_DB_SCHEMA_VERSION = 1`. No v1→v2 block
  anywhere. `no_commit_docker` is simply deleted with no migration.
- **Defensive `task_dir` fallback:** If the pinned `no_commit` mode says the task file should
  be in one location but it's actually in the other (hand-edited meta), `task_dir` returns the
  location where the file actually exists — preventing silent record loss.
- **`no_worktree` stays orthogonal.** It is an independent axis (chat, out-of-repo) and does
  not affect the commit mode.

### Patterns established

- **`SessionMeta` derived properties** (`record_dir`, `task_dir`, `docker_root`) centralize
  every "where does X live?" decision so call sites stay branch-free. All three key on the
  single `no_commit` field:
  - `task_dir` → `record_dir if self.no_commit else worktree_path/".hatchery"/"tasks"`, with a
    defensive fallback to the other location if the primary has no matching file.
  - `docker_root` → `docker_store_dir(...) if self.no_commit else (repo_path if no_worktree else worktree_path)`.
- **`ensure_repo_store(repo)`** creates `repos/<repo-id>/` with `records/` and `docker/` subdirs
  and writes/refreshes `repo.json` metadata. Idempotent.
- **`ensure_git_exclude(repo, entry)`** mirrors `ensure_gitignore` but targets `.git/info/exclude`
  (local, uncommitted) instead of the committed `.gitignore`.

### Gotchas for future agents

1. **`SessionMeta` uses `no_commit: bool = False`** (negative polarity). `SCHEMA_VERSION = 1`;
   no migration. The old `no_commit_docker` field is simply deleted (no migration).
2. **`write_task_file` and `find_task_file` signatures changed:** they now take `tasks_dir: Path`
   instead of `worktree: Path`. Use `meta.task_dir` to get the right directory.
3. **`session_prompt` signature changed:** now takes `(meta: SessionMeta, extra_note: str = "")`
   instead of `(name, worktree, extra_note)`.
4. **`update_task_file_status` signature changed:** takes `tasks_dir: Path` instead of `worktree: Path`.
5. **`resolve_runtime` signature changed:** takes `(docker_root, no_docker, backend=...)` instead of
   `(repo, worktree, no_docker, backend=...)`. Use `meta.docker_root`.
6. **`launch_sandbox_shell` takes `docker_root: Path | None = None`** — defaults to `repo` (committed
   mode). In not-committed mode the caller passes the store dir so the image builds from the store's
   Dockerfile.
7. **`ensure_docker_files_uncommitted` was removed.** In no-commit mode, `create()` writes docker files
   directly to `docker_store_dir(repo)` via `ensure_dockerfile`/`ensure_docker_config` (no `source=` copy).
   The `source=` parameter was also removed from both functions — they always generate from the built-in
   template.
8. **Two system prompt variants exist:** `_SESSION_SYSTEM` (commit mode) and `_SESSION_SYSTEM_NOCOMMIT`
   (no-commit mode). The no-commit variant says "stored outside the repo, not committed" instead of
   "merged into main."
9. **`cmd_sandbox` routes by `no_commit`** — committed mode (`not no_commit`) writes docker to `repo`
   and commits; not-committed mode (`no_commit`) writes to `docker_store_dir(repo)` and skips commit.
   `ensure_tasks_dir` is only called in committed mode.
10. **`task_dir` has a defensive fallback** — if the primary location (based on `no_commit` mode) has no
    `*-<name>.md` file but the other location does, it returns the other location. This prevents silent
    record loss when a task's pinned mode disagrees with the file's on-disk location.
