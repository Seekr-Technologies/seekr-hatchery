## Objective

currently, hatchery only has the concept of "tasks". All interactions require a task, task file, etc.

It would be useful to also have a non-task mode, where we can just start a claude/codex/whatever session in a sandbox.

- What should we call this subcommand?
- Presuambly, this only makes sense in the context of a sandbox (otherwise, why wouldn't you just use the agent directly)?
  - Do worktrees apply?

## Summary

### What was built
`hatchery chat [NAME]` — a lightweight mode that launches an agent in a Docker sandbox without task overhead. No worktree, no branch, no task file, no structured plan-implement-commit workflow, no post-exit menu.

### Key decisions
- **Docker-only**: Chat requires a git repo for Docker scaffolding. Without Docker it's just `claude` with extra steps.
- **Auto-naming**: `chat-1`, `chat-2`, ... — scans existing metadata for `type=chat` entries to find the next number.
- **`type` field in metadata**: Chat sessions store `"type": "chat"` in `meta.json`. Tasks implicitly have `type=task` (via `meta.get("type", "task")`). This field is shown in `hatchery status`.
- **Reuses sandbox_context()**: The existing `sandbox_context(no_worktree=True, branch="")` already produces a clean Docker-only description without task/branch references — no new function needed.
- **Simple post-exit**: "Mark this chat as complete? [Y/n]" — no wrap-up/delete/leave menu.
- **Full lifecycle support**: `resume`, `archive`, `delete`, `done` all work on chats.

### Patterns established
- The `is_chat` boolean parameter is threaded through `_launch_new` and `_launch_resume` as a keyword-only-style param (after `no_worktree`). This keeps the existing call sites unchanged while allowing chat to diverge on system prompt, initial prompt, banner, and post-exit behavior.
- The `type` field in metadata (`"task"` vs `"chat"`) is the canonical way to distinguish session types. Future session types could extend this pattern.

### Gotchas
- `_launch_finalize` (the wrap-up flow) is not chat-aware — chats don't have task files to finalize. If someone calls `hatchery done` on a chat, it goes through `_do_mark_done` which handles `no_worktree=True` gracefully (skips worktree removal).
- The Docker sandbox container could not be tested in CI without Python available in the container. All tests mock the Docker/subprocess layer.
