# Task: symlinks

**Status**: complete
**Branch**: hatchery/symlinks
**Created**: 2026-05-13 08:28

## Objective

Handle symlinks within the hatchery sandbox. Add a `follow_symlinks: false`
toggle to `docker.yaml`. When set to `true`, find symlinks in the mounted
working tree and bind-mount their external targets into the container so the
links resolve.

## Context

Users sometimes keep symlinks in their working tree pointing to absolute host
paths outside the worktree (shared data dirs, vendored caches, sibling repos).
Before this change the Docker sandbox only mounted the worktree (or cwd in
`--no-worktree` mode) plus `docker.yaml: mounts`, so those symlinks dangled
inside the container. `follow_symlinks: true` now scans the mounted area on
launch and bind-mounts each external symlink target at the same absolute host
path, so the link resolves identically inside the container.

## Summary

### Behavior

- `follow_symlinks: false` (default) — unchanged behavior.
- `follow_symlinks: true` — `_construct_symlink_mounts` walks the worktree
  (worktree mode) or cwd (no-worktree mode) with `os.walk(followlinks=False)`,
  resolves each symlink via `Path.resolve(strict=True)`, and emits
  `target:target:rw` mount strings for unique external targets.
- Skipped: targets already covered by an existing mount (or where an existing
  mount sits underneath the target), targets under the system blocklist
  (`/usr`, `/bin`, `/sbin`, `/lib`, `/lib64`, `/etc`, `/proc`, `/sys`, `/dev`,
  `/run`), broken/cyclic links, relative links staying inside the worktree.
- Walk prunes: `.git`, `.hatchery`, `node_modules`, `.venv`, `venv`,
  `__pycache__`, `.tox`, `.mypy_cache`, `.pytest_cache`.
- Mount mode is **read-write** so the agent can edit files reached through
  symlinks (matches user requirement).

### Key decisions

- **Same-path bind mount, not symlink rewriting.** The symlink's stored string
  is an absolute host path; mounting the resolved target at that exact path
  inside the container means the existing link resolves with no extra magic
  (e.g., shadowing each symlink file individually). Trade-off: this works only
  for absolute symlinks. Relative symlinks that escape the worktree still
  dangle (rare; documented as a v1 limitation).
- **System blocklist trimmed.** Initially included `/tmp` and `/var`, but
  those are common locations for user data (pytest fixtures live under `/tmp`),
  so they were removed. The blocklist now covers only paths whose contents are
  truly provided by the container image or kernel.
- **Sandbox shell (`launch_sandbox_shell`) intentionally not wired.** It
  mounts the full repo rw and is a quick-look tool; users wanting symlink
  support there can use explicit `mounts:` entries.
- **Chain limitation.** `Path.resolve(strict=True)` collapses symlink chains
  to the final real path, so only that path is mounted. A chain through
  another external symlink file won't fully resolve inside the container.
  Acceptable for v1; can be revisited if reported.
- **Two link shapes are rejected at launch, not silently dangled.** The
  worktree (or cwd) is mounted at a different absolute path inside the
  container than on the host, so:
  - absolute links whose target is *inside* the worktree, and
  - relative links whose target is *outside* the worktree,
  cannot resolve correctly in the container regardless of what we mount.
  `_construct_symlink_mounts` collects both lists during the walk and exits
  with an `ui.error` naming every offending link, telling the user to fix
  the link (rewrite to the other shape) or set `follow_symlinks: false`.
  The "fix-via-shadow-rewrite" alternative (per-link shadow files like the
  existing `worktree_git_ptr` mount) was deferred — error-first is honest
  and adds zero runtime complexity.

### Gotchas / notes for future agents

- The `_default_home_mounts()` helper returns ~/.gitconfig and the uv cache.
  When unit-testing mount assembly, monkeypatch it to `lambda: []` to avoid
  coupling tests to the developer's actual home directory.
- Tests use `pytest`'s `tmp_path`, which lives under `/tmp/...`. If you ever
  re-introduce `/tmp` to the system blocklist, those tests will silently
  produce empty mount lists and fail.
- `os.walk(followlinks=False)` still LISTS symlinked directories in
  `dirnames` (so `is_symlink()` picks them up) but does not recurse into
  them — the right knob for this scan.
- The helper takes `existing_mounts: list[str]` (the docker `-v` strings).
  Callers must pass the fully assembled list (after `_construct_docker_mounts`,
  `_default_home_mounts`, `backend.home_mounts(...)`) so dedup sees everything.
